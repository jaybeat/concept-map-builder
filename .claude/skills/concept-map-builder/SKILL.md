# 概念图构建器(Claude Code 版)

概念图(concept map,Novak)用**节点**表示概念、用**带文字的连线**表示概念间关系,两个概念加一个连接词构成一条**命题**。整张图是命题的网络,围绕一个**焦点问题**组织,自上而下从概括到具体,并尽量有跨分支的**交叉连接**。

本 skill 把"构建 → 评估 → 修订"做成闭环:六步搭出初稿,再起一个 evaluator 子代理按八项标准评分、给出修订项,据此改,必要时再评一轮。目标不是一次画完,而是收敛到一张经得起推敲的图。

> 本版本针对 Claude Code(终端 agent)。它**不依赖** claude.ai 的 `show_widget` / `read_me` / `ask_user_input_v0` 等交互工具;概念图写成**自包含 SVG 文件**,提问用纯文本,评估用真子代理。

## 输出形式

把概念图渲染成一个**自包含的 `.svg` 文件**写到磁盘(默认当前工作目录,文件名形如 `concept-map-<英文短名>.svg`;若用户给了路径就用它)。严格按 `references/svg-conventions.md` 的**自包含样式**来写——那里内嵌了具体颜色与字体,不依赖任何外部 CSS 变量或运行时类,直接用浏览器打开就有完整样式。

写完后只**报告文件的绝对路径**,并说一句可在浏览器打开。不要假设环境里有浏览器或 `open` 命令(可能是远程无显示的机器)。

## 第 0 步:判断输入是不是焦点问题

默认假设用户给的就是一个焦点问题,直接进入第 1 步。

如果输入是一个**大主题**(如"数据结构""操作系统")或一句不像问句的描述,**先不要画**。读 `references/focus-question.md`,从中挑 2–4 个最值得聚焦的候选焦点问题,**用纯文本列出来,然后停下,等用户回复选哪一个(或自己另给)**。一张图只回答一个焦点问题——替用户硬选,往往画歪。(本版没有 `ask_user_input_v0`,就直接在回复里编号列出候选并等待。)

## 六步构建

确定焦点问题后,按这六步走(每步都附"为什么",理解了再动手):

1. **确定焦点问题**。写在图顶部。好问题问"关系"(怎么/为什么/如何取舍/由什么决定),不是单纯"……是什么"——后者只得到一堆扁平定义。
2. **头脑风暴列概念(停车场)**。先把相关概念一股脑列出,**先列后连**,不要边想边纠结关系而卡住。
3. **从一般到具体排序**。最概括的在上,最具体的实例在下;太细或重复的先丢回停车场(一张图约 15–25 个概念为宜)。
4. **搭层级骨架**。先摆概念、拉线,不写关系,确认结构站得住。
5. **写连接词成命题**。每条线配**具体、有方向**的连接词,让"概念—连接词—概念"读起来是一句通顺、为真的话。禁用"相关/有关/包括"这类泛词。**命题方向必须为真**:别把度量/属性类概念(如"时间复杂度")当作被它度量之物的父节点。
6. **加交叉连接与实例**。主动找跨分支连线(交叉连接)——这是整合与创造力的体现,评估里**权重最高**。再给抽象概念挂上实例。

渲染时务必让具体需求**精确指向**具体概念(一一对应),避免一句连接词同时连向一片节点的"广播"式泛连。

## 评估(Claude Code 有子代理,用真子代理)

初稿写完后**不要直接收工**。起一个独立的 evaluator **子代理**(Task),让它读 `agents/evaluator.md`,只给它"焦点问题 + 概念图 **SVG 文件路径**",**不给构建思路**,以保持独立视角。它会按八项打分(交叉连接权重最高;正确性、目的性为一票否决项),并给出**按优先级排序、可直接执行**的修订项。

(注:本版默认有子代理。万一在不支持子代理的环境里运行,退化为自己"换上评审视角"做一次 fresh-eyes 自评,同样严格走完八项,并说明独立性稍弱。)

## 修订闭环

1. 按修订项改图(先处理一票否决项与交叉连接相关项)。
2. 重新生成 SVG 文件(覆盖原文件或写新版本),报告路径。
3. 若仍有高优先级问题,再评一轮——**最多两到三轮**。超过还理不顺,说明图太满,按 `references/focus-question.md` 拆成几张互链子图,而不是继续往一张里塞。

收敛后,简要说明这张图"答上焦点问题了吗",并提示可继续往某个节点下钻或拆子图。
